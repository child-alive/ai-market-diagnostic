## 第一部分 P0：品牌词 / 无品牌词分层核查（最高优先级，先做这个）

**背景**：GEO 行业测量学的第一原则——问题里带品牌名（"Deli 怎么样"）时 AI 提及品牌近乎必然，
测的只是品牌声誉与描述准确性；问题里不带品牌名（"墨西哥最好的文具品牌"）时 AI 仍主动推荐,
才是真实的推荐竞争力。两者混算会导致可见度虚高，是行业公认的 vanity metric 陷阱。

**任务**：
1. 核查 f141d182 真实运行：8 个问题中有几个是 tier=brand（或问题文本中含 Deli/得力及别名）？
   头版"50% 提及率、平均顺位 1.5"是否混入了品牌词问题？把核查结论写入 PROGRESS.md。
2. 无论核查结果如何，重构指标聚合：
   - 所有聚合指标（Visibility / SOV / AvgPosition / Citation）一律按 **branded / unbranded 分层**计算；
   - 报告头版核心数字只用 **无品牌词可见度（Unbranded Visibility）**；
   - 品牌词问题单独成板块，重新定位为"品牌认知与情感诊断"：AI 对品牌的描述是否准确、
     情感倾向、是否存在过时/错误信息。
3. 若头版洞察数字需要修正，同步修正执行摘要、report.html、方案说明中所有引用处。
4. 方案说明"独立判断"章节新增一条：解释为什么必须区分两类查询、混算的危害，
   注明这是 GEO 测量的行业共识（branded vs unbranded queries）。
5. 检查 question_gen 的 prompt：确保生成无品牌词问题时明确要求"不得包含品牌名及其别名"，
   品牌词问题单独定向生成。若当前 prompt 无此约束，修复并重跑一次小规模验证。

**指标数字诚实性附加要求**：头版所有数字必须可溯源到具体 run_id；绝对化表述一律收敛为
限定表述（如"官网无西语页面"→"在抓取范围内的 N 页中未发现西语内容 / 未发现 es-MX hreflang"）。

## 第四部分 P1：GEO 领域概念补齐（基于行业清单的反向审计）

以下是对照 2026 年 GEO 行业审计清单发现的缺口。原则：实现成本低的直接做进代码；
成本高的写入方案说明"下一步规划"章节（把未做的事变成路线图，本身就是成熟度信号）。

**A. site_audit 升级（直接实现，均为轻量检查）：**
1. AI 爬虫分项检查：解析 robots.txt，逐个报告主流 AI user-agent 的允许/屏蔽状态：
   GPTBot、OAI-SearchBot、ChatGPT-User、ClaudeBot、PerplexityBot、Google-Extended、
   Bytespider、CCBot。区分"训练爬虫"与"检索/引用爬虫"两类并在报告中分组展示。
2. llms.txt 检测：检查根目录是否存在 llms.txt / llms-full.txt（新兴标准，绝大多数网站没有，
   有无本身就是诊断亮点项）。
3. JS 渲染检查：对比原始 HTML 与页面主要内容——若正文依赖客户端 JS 渲染，
   AI 爬虫不可见。轻量实现：检查原始 HTML 中正文文本量/关键实体是否存在即可，不必上无头浏览器。
4. 内容可提取性启发式（可选）：首屏是否有直接回答式段落、是否存在 FAQ 结构。做不动就进路线图。

**B. 指标与文档层（低成本，写作为主）：**
5. "提及 vs 推荐"区分：AnswerAnalysis 增加 recommended 字段（AI 是否把品牌作为建议选项/首选，
   而不仅是列举），报告中 Mention Rate 与 Recommendation Rate 分开呈现。
6. 采样说明升级：方案说明中把"单次询问=噪声、正式测量需重复采样取分布"从注脚升为方法论小节；
   若额度允许，对 2~3 个核心问题做 3 次重复采样，展示波动区间作为方法论演示。
7. 来源质量分层：citations 按来源类型打标（官方站/权威媒体/电商平台/目录站/论坛），
   报告展示来源结构——这直接呼应"AI 引用全是第三方电商"的头版洞察。
8. 术语对齐：报告和文档统一使用行业术语——品牌词/无品牌词（branded/unbranded）、
   Share of Voice、Prompt Set、Query Fanout；适当处加一句 Share of Voice 与 Share of Answer 的
   区分脚注。不堆砌，用在刀刃上。

## 第六部分 P2：双受众演示网页 + 部署（时间充裕，正式立项）

**产品定位**：一个页面、两种模式。默认"回放模式"给产品/商务视角的人看故事；
"实况模式"给技术评审动手玩。这是前端能力展示位，质量标准对齐正式作品集。

**回放模式（默认，零 API 消耗，永不翻车）**：
- 数据源：f141d182 真实运行的完整产物（从 SQLite/JSON 读取，构建时打包为静态数据）；
- 演出效果：诊断管道逐阶段点亮（问题生成 → 多平台查询 → 回答分析 → 官网诊断 → 缺口 → 建议），
  西语问题逐条浮现、AI 回答流式滚动、指标卡片逐块拼装、最终汇成报告——
  呈现"多 Agent 调研"的临场感，但一切都是录制回放；
- 头版即洞察：用第一部分修正后的分层数字讲那个核心故事（AI 已在推荐、官网无承接）。

**实况模式（技术评审专用，显式开关进入）**：
- 迷你诊断：限定 3~5 个无品牌词问题的真实 DeepSeek 链路，流式展示进度；
- 防护：服务端频率限制（如每 IP 每小时 2 次）、并发上限 1、超时与失败的优雅降级
  （失败时提示并引导回放模式）；key 只存服务端。

**技术约束**：
- 前端 Vue 3 + TypeScript + Vite（呼应候选人技术栈）；构建产物为静态 dist；
- 后端复用现有管道，加一个极简 FastAPI 服务（仅实况模式需要）；
- 评审者通过线上 URL 访问，本地不需要 npm——仓库提交源码 + 构建产物，README 注明
  "在线访问为主，本地构建可选"；
- 资源画像适配 2 核 2G ECS：静态文件 + 单 worker FastAPI，无重依赖；
- 双受众切换：页面提供"产品视角/技术视角"切换，技术视角可展开每一步的原始 JSON、
  证据引用、等价 CLI 命令。

**部署（Fable5 产出物，用户执行）**：
- 输出一份 `deploy/部署手册.md`：Nginx/Caddy 静态托管 + FastAPI systemd 服务 +
  防火墙只开放所需端口 + 限流配置，按 IP:端口访问（不引入域名备案问题）；
- 提供一键部署脚本（scp/rsync + 远端安装），假定 Ubuntu/Debian 环境，脚本内不含任何密钥；
- 部署完成后把链接回填 SUBMISSION.md。

**红线**：演示网页是增强项，不得破坏现有 CLI 交付物；report.html 仍是提交主件；
网页任何一部分做不完就整体降级为"静态托管 report.html + 回放页"，不留半成品。
