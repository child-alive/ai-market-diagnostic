## 1. 我对需求的理解

这道题表面上是"做一个诊断工具"，实际上是出题方商业产品——**GEO（生成式引擎优化）诊断报告**——的一个原型切片。客户（如得力）真正想买的不是数据，而是三个问题的答案：

1. **我的目标市场用户在问 AI 什么？其中哪些问题值钱？**（问题发现 + 商业价值分级）
2. **用户问这些问题时，AI 推荐的是我还是竞品？依据是什么来源？**（AI 可见度：可见度、话语权、排名、引用、情感）
3. **AI 之所以不推荐我，我的官网/内容底座缺了什么？该先修什么？**（网站诊断 → 缺口 → 行动清单）

因此本原型的输出**刻意对齐 GEO 行业的指标语言**：Mention Rate / Visibility（提及率/可见度）、Recommendation Rate（推荐率）、SOV（Share of Voice 话语权）、Average Position（平均排名）、Citation Rate（来源覆盖率）、Sentiment（情感倾向）与 Source Mix（来源结构），问题按**品牌词 / 地区排名词 / 品类排名词**三层分类并标注 **TOFU/MOFU/BOFU** 漏斗——让报告读者（客户或顾问）拿到手就能用行业语言讨论，而不是面对一堆工程术语。

诊断的终点不是指标，是**决策**：报告最后一页必须能回答"先做哪三件事、为什么"。

## 2. 系统设计

### 2.1 数据流（六个独立模块 + 报告汇总）

```
BrandProfile(品牌/品类/市场/语言/官网/竞品种子)
  → ① question_gen     生成 20+ 条西语用户问题（三层分类+漏斗+价值分）
  → ② visibility_check 取 Top-8 高价值问题，按所选 Provider 分别获取"AI 回答"
  → ③ answer_analysis  各平台独立抽取提及/推荐/排名/竞品/来源类型/情感 → branded/unbranded 分层聚合
  → ③b query_fanout    可选：高价值无品牌词 → 3~5 个分支 → DeepSeek 抽样 → 独立 Coverage
  → ③c evidence_check  可选抓取来源页：回答陈述 → 页面证据片段 → 机器预审状态
  → ④ site_audit       官网轻量诊断：AI crawler/llms.txt/原始 HTML/robots/sitemap/hreflang/JSON-LD
  → ⑤ gap_analysis     综合 ①②④ → 内容缺口清单（每条附证据链与关联问题）
  → ⑥ recommendations  P0/P1/P2 行动清单（行动+理由+预期影响+工作量）
  → ⑦ report           DiagnosticReport(JSON) → report.html → SQLite runs 存档
  → ⑧ demo             固化 JSON 回放（产品/技术视角）/ 受限 FastAPI SSE 实况（不回写主 run）
```

模块间只通过 `src/models.py` 中的 Pydantic 契约传递数据，每个模块可独立运行与测试。

### 2.2 关键工程决策

| 决策 | 理由 |
| --- | --- |
| **AnswerProvider 抽象层**（DeepSeek / OpenAI Search / Gemini Grounding / Mock） | 三个平台使用各自联网搜索接口，同一问题地图可独立采样、独立计算指标；未配置 Key 的平台不运行，不伪装成真实结果 |
| **无 key 自动全 Mock** | 评审者不配置任何东西也能 `python -m src.main --mock` 跑通全链路；fixtures 经过分层设计（5 条无品牌/2 条排名靠后/1 条排名前列），保证演示结论有层次 |
| **Mock 三层标注** | 数据层 `is_mock` 字段、报告层 MOCK 角标、文档层本章节——模拟数据是题目允许的，清晰的取舍解释本身是交付物 |
| **结构化抽取双路径** | hybrid 模式用 DeepSeek JSON-mode 抽取，失败重试 1 次后降级到启发式（词典+正则）并标记 `parse_degraded`；mock 模式直接走启发式，确定性、可单测 |
| **DeepSeek V4 原生 Web Search** | 回答阶段通过 Anthropic 兼容端点调用服务端搜索工具，直接提取 `web_search_tool_result` 中的来源 URL；搜索失败时明确降级、不伪造 URL |
| **引用证据分层** | Citation Rate 只回答“API 是否返回搜索来源”；可选证据预审再抓取来源页并比较回答陈述与页面片段，输出支持状态且固定标记需人工复核 |
| **品牌词 / 无品牌词分层** | 品牌词衡量 AI 对品牌的认知、描述和情感；无品牌词衡量主动推荐竞争力。头版只展示 Unbranded Visibility，避免把“问题已给出答案”误当成主动推荐 |
| **提及 / 推荐分开** | Mention 只表示品牌进入答案；Recommendation 要求品牌被明确建议或进入“推荐/最佳选项”列表。保守启发式避免把客观介绍冒充购买建议 |
| **来源结构分层** | 引用按目标品牌官网、权威媒体/政府、电商、目录、论坛和其他分类；它回答“AI 从哪里形成叙事”，不把域名类型冒充逐句事实核验 |
| **Query Fanout 独立切片** | 只从高价值无品牌词派生同义改写、场景细化与追问式分支；生成后再检查品牌/别名泄漏，分支 Coverage 不混入主 Prompt Set，避免扩大样本后悄悄改变主指标 |
| **AI 访问能力分项** | 不把 robots.txt 汇总成单一“可抓取/不可抓取”：分别检查训练、搜索/引用与 Grounding 相关 token；`llms.txt`、JS 依赖、直接回答段落和 FAQ 仅作为启发式信号 |
| **缺口与建议规则驱动** | 由指标和证据推导而非再问一次 LLM——保证报告数据自洽、每条建议可溯源，也省额度 |
| **完整报告 JSON 存 SQLite** | `runs` 表以 `run_id` 为主键保存每次运行的 `DiagnosticReport`；历史报告可脱离 Provider 与网络重渲染，便于复盘与后续做趋势对比 |
| **静态 HTML 报告** | 蓝白专业风格，中西双语标签；无外部依赖的 CSS 图表展示竞品 SOV，评审者离线双击文件也能阅读全部结论 |
| **双受众网页与主链路隔离** | Vue 回放只读 `37b442ec`，产品视角讲洞察、技术视角展开 JSON/来源/命令；实况只允许固定 3~5 个无品牌词并用 SSE 流式展示，不把临场样本写回主指标 |
| **实况子进程而非后台线程** | 每 IP 每小时 2 次、全局并发 1；上游请求运行在独立 Python 子进程，总超时会终止整个进程，避免 HTTP 已提示失败但线程仍在后台继续消耗额度 |

### 2.3 指标口径

除了为兼容旧报告保留顶层全样本字段，以下指标均分别对 **branded / unbranded** 查询独立计算，不互相混入。问题只要是 `tier=brand`，或其西语/中文文本出现品牌名及别名，就归入 branded；这能容错 LLM 将“在哪里买 Deli”误标为 regional。

- **Mention Rate / Visibility** = 品牌出现的回答数 / 同口径的检测回答数；报告头版只使用 **Unbranded** 切片；
- **Recommendation Rate** = 品牌被明确建议或进入推荐候选列表的回答数 / 同口径的检测回答数。普通事实描述、历史背景和渠道陈列只算 Mention，不算 Recommendation；
- **SOV** = 品牌的回答级提及数 / 全部品牌（含竞品）的回答级提及数——每条回答里每个品牌至多计 1 次，避免单条回答刷屏；零售商/电商平台（Lumen、Office Depot、Amazon 等）不计入品牌 SOV，它们是渠道不是竞品；
- **Share of Voice ≠ Share of Answer**：本报告前者计算回答级品牌提及份额；后者通常指答案文本篇幅/字符份额，本原型不计算，避免同名术语混用；
- **Avg Position** = 品牌在其被提及回答中的平均顺位（按品牌在回答中出现次序）；
- **Citation Rate（来源覆盖率 Source Coverage）** = 至少带有一个 Web Search 来源 URL 的回答数 / 检测回答数；URL 由对应平台 API 返回。它衡量“回答是否附带可验证来源”，**不是“品牌被引用的比例”**，也不衡量“来源是否支持陈述”。报告显示名统一为“来源覆盖率”，以避免与品牌提及类指标混淆——因此“无品牌词提及率 0% 而来源覆盖率 100%”并不矛盾：AI 检索了大量来源作答，只是这些回答与来源都没有出现目标品牌；
- **Evidence Support Rate（证据预审支持率）** = 词面预审为“支持”的可评估陈述数 / 可评估陈述数。OpenAI/Gemini 优先使用原生引用文本区间；DeepSeek 没有区间时拆分答案陈述并在受限来源池中寻找最佳页面片段。无法访问和未映射单列，不混入分母；所有结果均需人工复核；
- **Sentiment** = 回答对目标品牌的倾向计数（正/中/负）。
- **Source Mix** = 同口径引用条目按目标品牌官网 / 权威媒体与政府 / 电商 / 目录 / 论坛 / 其他的数量结构。分类由透明域名规则产生，不等同于来源质量评分或内容支持结论。
- **Branch Mention / Recommendation Coverage** = Query Fanout 已检测分支中提及 / 明确推荐品牌的比例；**Parent Fanout Coverage** = 至少一个分支提及品牌的父问题数 / 已选父问题数。它们衡量问题分支中的覆盖深度，不代表搜索量，也不混入主 Prompt Set 的 Visibility。

### 2.4 错误处理与可复现性

- DeepSeek 请求失败会重试 1 次；Web Search 仍失败时降级为明确无联网的普通回答且不伪造 URL，结构化抽取失败则回退启发式解析并在 `parse_degraded` 中留痕；
- 官网请求受限或超时时自动读取本地快照，`snapshot_mode=true`，报告明确显示“快照模式”；
- 每条模拟回答均有 `is_mock=true`；报告对全局数据、派生指标、问题检测结果和回答明细逐层标注 Mock；
- 每次新诊断自动写入 `data/diagnostic.db`，可用 `python -m src.main --run-id <ID>` 恢复原始生成时间与完整数据后重渲染；无效 ID 会返回明确错误。
- 页面证据预审默认关闭；显式传 `--verify-evidence` 后才抓取公开来源页，并用 `--evidence-max-sources` / `--evidence-max-claims` 控制请求规模。

### 2.5 重复采样不是附注，而是测量方法

LLM 回答和检索结果具有随机性，单次询问只能作为一次观测。正式测量应固定 Prompt Set、平台、地区、语言、模型与采样窗口，对每题重复请求，分别保存每轮回答与来源，再报告均值、波动范围和足量样本下的置信区间；不能把单轮 0% 或 100% 当成稳定市场事实。

本原型提供独立命令 `python -m src.sampling_demo` 做最小演示，不改动主报告口径。2026-07-17 对 q05/q07/q08 三个高价值无品牌词各运行 3 次真实 DeepSeek Web Search：**9/9 成功且带搜索来源**。三轮 Prompt Set 的 Mention Rate 与 Recommendation Rate 均在 **0%~33.3%** 间波动；q08 只在第 2 轮提及并推荐 Deli，顺位为第 15，另两轮未提及。完整原文、61 个本轮返回来源 URL、逐轮判断与失败列表保存在 `data/示例产物/repeat_sampling.json`。这证明单次值有噪声，但 3×3 仍只是方法演示，不是统计置信区间。

## 3. 范围与取舍（为什么先做这些）

**做了：** 上述完整诊断闭环、得力×墨西哥演示用例、官网轻量实抓与快照降级、SQLite 历史运行存档、DeepSeek/OpenAI/Gemini 多平台数据切片、Query Fanout、可选来源证据预审、可直接查看的静态 HTML 报告，以及与主链路隔离的双受众回放/实况网页。

**刻意不做：**

- **跨平台共识缺口规则** —— 多平台已独立采样和展示，但内容缺口与建议仍以主平台切片生成，避免在没有多轮真实数据前擅自混合不同平台的指标；
- **大规模爬虫** —— 官网诊断限 15 页、限速 1 req/s，够回答"官网是否适合 AI 抓取"即可，抓更多页不改变诊断结论；
- **正式数据库 / 登录** —— 单机 SQLite + 静态 HTML 已满足主交付的“可查看、可迭代”；冲刺演示网页与主 CLI 隔离，不把作品集增强项冒充生产级多租户系统；
- **把网页变成唯一入口** —— 报告是“发给客户的文件”，静态 HTML 双击即开仍是最低摩擦的提交主件；网页只作为回放和技术演示层。

### 3.1 得力官网实测结果（2026-07-16）

在 `--mock --live-audit` 模式下，AI 回答部分保持可复现的 Mock，网站诊断则对 `deliworld.com` 实时抓取 15 页。结果如下：

| 检查项 | 结果 | 判断 |
| --- | --- | --- |
| robots.txt / 可抓取性 | 通过 | 基础抓取入口正常 |
| sitemap | 通过 | URL 发现机制正常 |
| JSON-LD 结构化数据 | 通过 | 已有一定机器可理解基础 |
| `es-MX hreflang` | **15 页抓取范围内未发现** | 当前样本不能向搜索与 AI 系统表明墨西哥西语版本 |
| 西班牙语内容 | **15 页抓取范围内未发现** | 样本提示目标市场内容承接不足，不代表全站结论 |

本次检查发现的现有 hreflang 为 `en / en-EU / ko / ru / vi / zh-Hans-CN / zh-Hant-TW`。因此报告将 `NO_HREFLANG_ES_MX` 与 `NO_SPANISH_CONTENT` 评为 high，并把两项技术/内容信号直接转成 P0 修复建议。该结果是一次小样本技术扫描，不等价于完整 SEO 审计；提交示例的 `report.json` 保留了本次实抓结果，`snapshot_mode=false`。

2026-07-17 使用新增检查再次运行 `--mock --live-audit`（`run_id=a7ec0a37`，AI 回答为 Mock、官网为实时抓取）：15 页范围内的 robots.txt 规则允许本原型检查的 8 个 AI 访问 token；根目录未发现有效的 `llms.txt` / `llms-full.txt`；首页原始 HTML 可提取约 24,824 个正文字符且包含品牌实体，未触发“可能依赖客户端 JS”的预警；发现直接回答式段落，未发现 FAQ 结构。以上只描述该次抓取与启发式结果，不证明 WAF/CDN 实际放行，也不等价于浏览器渲染后的完整 DOM 审计。

爬虫用途按官方口径记录：OpenAI 将 OAI-SearchBot 与 GPTBot 分别用于搜索可发现性和潜在训练控制；Google-Extended 是同时影响 Gemini 训练与部分 Grounding 的 robots.txt 控制 token，并非独立 HTTP user-agent；PerplexityBot 与 ClaudeBot 也按各自官方说明归类。参考：[OpenAI Publishers FAQ](https://help.openai.com/en/articles/12627856-publishers-and-developers-faq)、[Google crawler 文档](https://developers.google.com/crawling/docs/crawlers-fetchers/google-common-crawlers)、[Perplexity crawler 文档](https://docs.perplexity.ai/docs/resources/perplexity-crawlers)、[Anthropic crawler 说明](https://support.claude.com/en/articles/8896518-does-anthropic-crawl-data-from-the-web-and-how-can-site-owners-block-the-crawler)。

最终提交示例 `run_id=37b442ec` 完成了 `deepseek-v4-flash` + 原生 Web Search 主链路与 Query Fanout 的同轮真实验收：实时生成 22 条主问题、回答 Top-8、完成 8 条 JSON-mode 结构化抽取，全部 `is_mock=false`、`search_grounded=true` 且无解析降级，主回答共返回 70 个来源 URL。Top-8 实际为 **3 个 branded + 5 个 unbranded**：核心数字为 **Unbranded Mention Rate / Recommendation Rate / SOV 均为 0%、Average Position 无、来源覆盖率（Citation Rate）100%**；branded 切片为 Visibility 100%、Recommendation Rate 66.7%、Average Position 1.0，3 条情感均为正面。

对该 run 的主回答来源做透明域名规则分层后，整体 Source Mix 为：目标品牌官网 8、权威媒体/政府 7、电商 5、目录 5、论坛 2、其他 43；其中 5 个无品牌词回答的 37 个来源中，**目标品牌官网为 0**，权威媒体/政府 3、目录 5、其他 29。它支持一个谨慎的商业判断：本轮无品牌词回答没有主动提及 Deli，目标品牌官网也没有进入无品牌词来源池，品牌尚未控制这些需求场景中的 AI 叙事与承接链路。来源类型仍不证明页面逐句支持回答。

同一运行还选取 q01、q02 两个价值分 4 的无品牌词父问题，各生成同义改写、场景细化、追问式 3 个分支；6 个问题均未包含 Deli/得力或别名，6/6 由 DeepSeek Web Search 成功回答并返回 66 个来源 URL。Parent Fanout Coverage、Branch Mention Coverage 与 Branch Recommendation Coverage 均为 0%。因此“未进入通用需求答案”的观测不仅出现在主 Prompt Set，也出现在本次分支样本；但两者仍是同平台、同日的小样本，不构成跨平台独立验证或统计置信区间。

双受众网页在 2026-07-17 由内置浏览器完成真实实况验收：固定 q05 / q07 / q06 三个高价值无品牌词，逐题 SSE 流式显示，3/3 `search_grounded=true`，分别返回 3 / 7 / 10 个来源 URL，三题均未提及 Deli，最终状态为 `DONE`。这次 3 题只验证服务端 Key 隔离、固定 Prompt Set、Web Search 与前端流协议，不回写 `37b442ec`，也不与主报告或 Query Fanout 指标混算。

历史运行 `f141d182` 保留为测量口径修正的审计记录：其 Top-8 实际为 4 branded + 4 unbranded，旧口径的 50% Visibility 和 1.5 Average Position 全部来自含品牌词问题；修正后无品牌词 Mention / Recommendation / SOV 同样为 0%。新提交示例已由 `37b442ec` 取代它，以便把分层指标、进阶站点审计和 Query Fanout 放在同一个可追溯 run 中。以上结论都只代表 DeepSeek 的本轮观测，不代表 ChatGPT/Gemini 在墨西哥市场的真实表现。

## 4. 限制与不确定性（独立判断）

1. **品牌词不能代替无品牌词可见度**：区分 branded vs unbranded queries 是 GEO 测量的行业共识。问题本身已给出品牌名时，AI 复述该品牌几乎是题意要求；把这类回答与通用品类推荐混算，会制造虚高的 vanity metric，令团队误判真实竞争力和投入优先级。因此品牌词只用于认知/情感/描述准确性诊断，头版只用无品牌词指标。
2. **AI 回答不可复现**：3 题×3 轮真实演示已经观察到 Prompt Set 的 Mention/Recommendation Rate 在 0%~33.3% 波动，单次采样 ≠ 稳定可见度。正式产品仍需扩大 Prompt Set 与轮数后报告均值、分布和置信区间；本次小样本不能计算可靠置信区间。
3. **平台覆盖限制**：OpenAI Search 与 Gemini Google Search Grounding 的 Provider 代码和本地契约测试已完成，也用本机配置发起过最小真实请求；但 OpenAI 因 API 账户无可用额度返回 429，Gemini 的 Google Search Grounding 因额度/权限返回 429，因此两者均不能声称联网真实链路已通过。当前提交示例仍只代表 DeepSeek；DeepSeek 的回答分布与 ChatGPT/Gemini 在墨西哥市场的真实分布**不等价**。
4. **样本量限制**：8~10 个问题是演示规模。商业结论需要 100+ 问题、多城市多语言变体采样（墨西哥西语 ≠ 西班牙西语）。
5. **抓取伦理与限制**：遵守 robots.txt、限速、仅抓公开页面；对方反爬时降级为快照模式并在报告标注。
6. **LLM 结构化抽取存在误差**：品牌别名（Deli/得力/DeliMex）与西语变体可能漏检；启发式词典只覆盖常见品牌。生产环境应加实体词典服务与人工抽检校准。
7. **"商业价值评分"是模型主观判断**：正式产品应结合 Google Trends/搜索量/转化数据校准，当前 1~5 分仅用于排序优先级，不宜对外报价使用。
8. **搜索来源不等于逐句已核验**：Citation Rate 只说明回答获得了平台搜索来源。可选证据预审会进一步抓取页面、保存最佳证据片段并标记支持/部分支持/未找到/无法访问，但当前算法只做可复现的词面覆盖，无法可靠识别语义改写、时间失效与上下文矛盾；它是人工核验队列的排序工具，不是人工审核的替代品。
