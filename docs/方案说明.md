## 1. 我对需求的理解

这道题表面上是"做一个诊断工具"，实际上是出题方商业产品——**GEO（生成式引擎优化）诊断报告**——的一个原型切片。客户（如得力）真正想买的不是数据，而是三个问题的答案：

1. **我的目标市场用户在问 AI 什么？其中哪些问题值钱？**（问题发现 + 商业价值分级）
2. **用户问这些问题时，AI 推荐的是我还是竞品？依据是什么来源？**（AI 可见度：可见度、话语权、排名、引用、情感）
3. **AI 之所以不推荐我，我的官网/内容底座缺了什么？该先修什么？**（网站诊断 → 缺口 → 行动清单）

因此本原型的输出**刻意对齐 GEO 行业的指标语言**：Visibility（可见度）、SOV（Share of Voice 话语权）、Average Position（平均排名）、Citation Rate（引用率）、Sentiment（情感倾向），问题按**品牌词 / 地区排名词 / 品类排名词**三层分类并标注 **TOFU/MOFU/BOFU** 漏斗——让报告读者（客户或顾问）拿到手就能用行业语言讨论，而不是面对一堆工程术语。

诊断的终点不是指标，是**决策**：报告最后一页必须能回答"先做哪三件事、为什么"。

## 2. 系统设计

### 2.1 数据流（六个独立模块 + 报告汇总）

```
BrandProfile(品牌/品类/市场/语言/官网/竞品种子)
  → ① question_gen     生成 20+ 条西语用户问题（三层分类+漏斗+价值分）
  → ② visibility_check 取 Top-8 高价值问题，逐条获取"AI 回答"（Provider 抽象）
  → ③ answer_analysis  每条回答结构化抽取：品牌提及/排名/竞品/引用/情感 → 聚合五大指标
  → ④ site_audit       官网轻量诊断：robots/sitemap/hreflang/JSON-LD/西语内容
  → ⑤ gap_analysis     综合 ①②④ → 内容缺口清单（每条附证据链与关联问题）
  → ⑥ recommendations  P0/P1/P2 行动清单（行动+理由+预期影响+工作量）
  → ⑦ report           DiagnosticReport(JSON) → 自包含 report.html
```

模块间只通过 `src/models.py` 中的 Pydantic 契约传递数据，每个模块可独立运行与测试。

### 2.2 关键工程决策

| 决策 | 理由 |
| --- | --- |
| **AnswerProvider 抽象层**（DeepSeekProvider + MockProvider） | 国内网络无法直连 ChatGPT/Gemini API，用 DeepSeek 扮演"AI 回答引擎"角色；接入真实海外平台只需新增一个实现类，pipeline 零改动 |
| **无 key 自动全 Mock** | 评审者不配置任何东西也能 `python -m src.main --mock` 跑通全链路；fixtures 经过分层设计（5 条无品牌/2 条排名靠后/1 条排名前列），保证演示结论有层次 |
| **Mock 三层标注** | 数据层 `is_mock` 字段、报告层 MOCK 角标、文档层本章节——模拟数据是题目允许的，清晰的取舍解释本身是交付物 |
| **结构化抽取双路径** | hybrid 模式用 DeepSeek JSON-mode 抽取，失败重试 1 次后降级到启发式（词典+正则）并标记 `parse_degraded`；mock 模式直接走启发式，确定性、可单测 |
| **缺口与建议规则驱动** | 由指标和证据推导而非再问一次 LLM——保证报告数据自洽、每条建议可溯源，也省额度 |

### 2.3 指标口径

- **Visibility** = 品牌出现的回答数 / 检测回答数；
- **SOV** = 品牌的回答级提及数 / 全部品牌（含竞品）的回答级提及数——每条回答里每个品牌至多计 1 次，避免单条回答刷屏；零售商/电商平台（Lumen、Office Depot、Amazon 等）不计入品牌 SOV，它们是渠道不是竞品；
- **Avg Position** = 品牌在其被提及回答中的平均顺位（按品牌在回答中出现次序）；
- **Citation Rate** = 含可溯源引用（域名）的回答占比；
- **Sentiment** = 回答对目标品牌的倾向计数（正/中/负）。

## 3. 范围与取舍（为什么先做这些）

**做了：** 上述完整单链路（题目第七部分点名的最小闭环），得力×墨西哥演示用例，自包含 HTML 报告。

**刻意不做：**

- **多平台并发检测** —— 平台差异被 Provider 抽象层吸收，多平台只是横向复制，不新增认知价值；先证明单链路端到端正确；
- **大规模爬虫** —— 官网诊断限 15 页、限速 1 req/s，够回答"官网是否适合 AI 抓取"即可，抓更多页不改变诊断结论；
- **正式数据库 / 登录 / 部署** —— SQLite + 静态 HTML 已满足"可查看、可迭代"，部署会抬高评审成本；
- **前端框架** —— 报告是"发给客户的文件"而非"运营的系统"，自包含 HTML 双击即开，是该场景下交付价值最高的形态。

## 4. 限制与不确定性（独立判断）

1. **AI 回答不可复现**：同一问题多次询问结果不同，单次采样 ≠ 稳定可见度。本原型单轮采样用于演示；正式产品需多轮采样取均值并给出置信区间。
2. **平台覆盖限制**：国内网络环境下 ChatGPT/Gemini API 不可直连。本原型用 DeepSeek 代理"AI 回答引擎"角色并做了 Provider 抽象；DeepSeek 的回答分布与 ChatGPT/Gemini 在墨西哥市场的真实分布**不等价**，结论方向可参考、数值不可直接对客户引用。
3. **样本量限制**：8~10 个问题是演示规模。商业结论需要 100+ 问题、多城市多语言变体采样（墨西哥西语 ≠ 西班牙西语）。
4. **抓取伦理与限制**：遵守 robots.txt、限速、仅抓公开页面；对方反爬时降级为快照模式并在报告标注。
5. **LLM 结构化抽取存在误差**：品牌别名（Deli/得力/DeliMex）与西语变体可能漏检；启发式词典只覆盖常见品牌。生产环境应加实体词典服务与人工抽检校准。
6. **"商业价值评分"是模型主观判断**：正式产品应结合 Google Trends/搜索量/转化数据校准，当前 1~5 分仅用于排序优先级，不宜对外报价使用。

## 5. 完成/未完成边界与下一步

**已完成：** 全链路可运行（mock 与 hybrid 双模式）、五大指标聚合、带证据链的缺口与 P0/P1/P2 建议、自包含 HTML 报告、Mock 三层标注。

**未完成/进行中：** 官网真实抓取诊断（当前为快照占位，Stage 2）、SQLite 历史运行存档、报告视觉升级与图表、单元测试补齐。

**下一步（按优先级）：**
1. site_audit 真实抓取（robots/sitemap/hreflang/JSON-LD/西语内容检测）+ 降级快照；
2. 报告视觉升级（指标卡片、ECharts 竞品对比图）与测试补齐；
3. 第二个真实 Provider（Gemini/SERP API）做跨平台对比；
4. Query Fanout：高价值问题自动派生 3~5 个子问法抽样检测，度量"问题分支"覆盖深度；
5. 多轮采样与置信区间，把演示指标升级为可对客户交付的指标。
